# Requirement Confirmation

通用需求确认门禁：

1. 需求分析结束后，不允许默认直接切到 UI 设计师或交付阶段
2. 只有当 `用户确认状态 = 允许进入下一阶段` 时，才允许进入 UI 设计或交付
3. 只要缺口会影响页面形态、信息架构、内容来源、主 CTA、核心交互或关键边界态，就必须先问用户
4. 需求分析输出应控制为：
   - 3-5 行已知摘要
   - 1-3 个最关键待确认问题
5. 没有拿到回答前，不要输出完整方案，不要切下一个阶段
6. 用户只回复“确认 / 可以 / 按这个来 / 没问题”时，只能确认上一轮明确摘要中的条目
7. 如果仍有摘要外的未决项，必须先回写“已确认 / 待确认”，不能把一句确认扩展成全部通过

提问预算补充：

- `L1`：0-2 问，通常不进入完整确认门禁
- `L2`：2-4 问，确认项只保留会改变页面结构或交互的内容
- `L3`：4-7 问，允许覆盖流程、异常、边界和重设计判断
- `L4`：6-10 问，但必须分阶段确认，不一次问完

确认门禁追求的是“够用收口”，不是“问题越多越安全”。
