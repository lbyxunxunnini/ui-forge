# Recipes

这些 recipes 用于把常见 UI 任务快速收口，不替代主流程。

## 1. 页面重设计

适用：

- 现有页面能用，但像模板
- 信息密度失衡
- 视觉语言不统一

顺序：

1. 先诊断，不先画新稿
2. 指出 1 个整体问题和 2-3 个局部问题
3. 给出新的 layout thesis
4. 再输出组件、tokens 和交付稿

## 2. 后台/仪表盘

重点：

- 先定密度，再定风格
- 一屏只保留一个主焦点
- 侧栏、筛选、表格、状态标签必须属于同一视觉语言

禁止：

- 大圆角卡片铺满全屏
- 每块数据都上阴影
- 头部、筛选区、表格同时用强背景

## 3. 登录/注册流

重点：

- 先确定路径数量，再设计视觉
- 登录、验证码、第三方登录、找回密码的优先级必须明确
- 交互状态要完整：focus、error、disabled、loading、success

禁止：

- 只画静态默认态
- 为了“高级感”牺牲表单可读性

## 4. 设计系统收口

重点：

- 先收口 type scale 和 spacing scale
- 再定义 semantic tokens
- 最后才是组件样式

禁止：

- 从按钮长相倒推整套系统
- token 很规范但组件层没有主张

## 5. 评审模式

输出顺序：

1. 整体诊断
2. 高优先级问题
3. 修正建议
4. 不建议做的事

只有在用户明确要实现稿时，才进入 HTML/CSS/SVG 交付。
