# Evaluation Rubric

用这份 rubric 评估 `ui-forge` 的输出是否真的比普通 UI prompt 更强。

## 评分维度

每项 0-2 分，总分 10 分。

### 1. 需求收口

- `0` 关键约束缺失，还在脑补
- `1` 收口了显性需求，隐性约束不足
- `2` 平台、流程、异常、状态、边界都已明确

### 2. 结构判断

- `0` 直接堆组件
- `1` 有基本布局，但没有明确主次
- `2` 有清晰的信息层级、焦点和浏览路径

### 3. 视觉语言

- `0` 泛模板化，默认审美明显
- `1` 有风格选择，但一致性不足
- `2` 风格明确，type、space、surface、color 说同一种语言

### 4. 交互完整性

- `0` 只覆盖静态态
- `1` 覆盖常见状态
- `2` 覆盖 focus、error、empty、loading、success、responsive

### 5. 交付可用性

- `0` 只有概念，没有落地
- `1` 有部分落地材料
- `2` 有可直接交给前端或后续 LLM 的设计说明和产物结构

## 及格线

- `8/10`：可用于真实项目试跑
- `9/10`：可作为开源展示样例
- `< 8/10`：需要回到结构判断或交互完整性修正

## Baseline 对比

评测时至少对比两组输出：

- 普通“帮我设计一个页面” prompt
- `ui-forge` 输出

观察差异时，优先看：

- 有没有先收口再设计
- 有没有解释为什么这样设计
- 有没有明确禁止项
- 有没有交互和状态层
