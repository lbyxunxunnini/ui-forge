# 输入不完整处理

## 核心原则

1. **不要假装看懂设计图**：如果用户提供了设计图，但无法清晰理解，必须向用户确认
2. **不要自动补齐需求空白**：不能自动补默认值，必须向用户确认
3. **先理解再执行**：在理解需求之前，不要急于执行

## 输入类型处理

### 文字描述

- 如果描述清晰，直接执行
- 如果描述模糊，向用户确认关键点
- 如果缺少必要信息，向用户补充

### 设计图

- 如果能清晰理解，直接执行
- 如果无法理解，向用户确认
- 如果部分理解，先确认不确定的部分

### PRD/需求文档

- 如果文档完整，直接执行
- 如果文档不完整，向用户补充缺失部分
- 如果有歧义，向用户确认

## 确认机制

### 确认时机

1. 需求理解后
2. 关键决策点
3. 方案确认前

### 确认格式

```
[ui-forge] 需求分析师：需要确认
- 确认点：（具体问题）
- 选项A：（选项描述）
- 选项B：（选项描述）
- 推荐：（推荐选项）
```

### 确认规则

1. 确认点必须具体，不能模糊
2. 必须给出选项，不能只问开放性问题
3. 必须给出推荐，不能只问用户
4. 用户确认后，才能继续执行

## 不完整输入处理

### 缺少平台信息

```
[ui-forge] 需求分析师：需要确认
- 确认点：目标平台是什么？
- 选项A：iOS应用
- 选项B：Android应用
- 选项C：Web网页
- 选项D：响应式设计（多端适配）
- 推荐：根据描述推测，推荐选项X
```

### 缺少页面类型

```
[ui-forge] 需求分析师：需要确认
- 确认点：需要设计什么页面？
- 选项A：登录/注册页
- 选项B：首页/Dashboard
- 选项C：个人中心/设置页
- 选项D：其他页面（请描述）
- 推荐：根据描述推测，推荐选项X
```

### 缺少风格信息

```
[ui-forge] 需求分析师：需要确认
- 确认点：设计风格是什么？
- 选项A：简约风格（干净、留白）
- 选项B：现代风格（渐变、卡片）
- 选项C：商务风格（专业、稳重）
- 选项D：科技风格（深色、霓虹）
- 选项E：活泼风格（彩色、圆润）
- 推荐：根据描述推测，推荐选项X
```

### 缺少功能信息

```
[ui-forge] 需求分析师：需要确认
- 确认点：需要包含哪些功能？
- 功能列表：
  1. 表单输入（手机号、密码）
  2. 社交登录（微信、Apple）
  3. 动画效果（渐变背景）
  4. 其他功能（请补充）
- 推荐：根据描述推测，推荐功能X
```

## 返回机制

### 返回格式

```
[ui-forge] 页面工程师：返回 UI 设计师
- 缺失内容：按钮点击后的加载状态和成功/失败反馈
- 请补充：加载中的样式、成功后的跳转逻辑、失败时的提示方式
```

### 返回规则

1. 返回只允许向上游返回，不允许跨角色返回
2. 返回后，被返回角色必须响应，不允许忽略
3. 响应格式与返回格式相同，只是把"缺失内容"改为"补充内容"
4. 一轮任务最多返回 2 次。超过 2 次仍未解决，由决策权最高的角色拍板
